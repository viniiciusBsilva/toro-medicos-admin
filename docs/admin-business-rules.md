## 3) Tabelas de Autenticação e Permissão do Admin

### 3.1 `public.user_admin`

Registra quais usuários têm acesso ao painel administrativo. Vincula ao `public.user` existente, reutilizando o mesmo sistema de autenticação do Supabase Auth.

```sql
create table public.user_admin (
  created_at timestamp with time zone not null default now(),
  id_user    uuid null,
  id         uuid not null default gen_random_uuid(),
  constraint user_admin_pkey        primary key (id),
  constraint user_admin_id__key     unique (id),
  constraint user_admin_id_user_fkey
    foreign key (id_user) references "user" (id)
    on update cascade on delete cascade
);
```

**Colunas:**
| Coluna | Tipo | Descrição |
|---|---|---|
| `id` | uuid PK | Identificador do registro admin |
| `id_user` | uuid FK → user.id | Referência ao usuário base (Supabase Auth) |
| `created_at` | timestamptz | Data de criação do acesso |

**Regras:**
- Um usuário só pode ter **um** registro em `user_admin`
- A exclusão em `user` remove em cascata o registro em `user_admin` e `admin_permissoes`
- Ao convidar novo admin: criar via Supabase Auth invite → inserir em `public.user` → inserir em `user_admin`
- Ao revogar acesso: deletar de `user_admin` (cascata remove permissões; não deletar o `user`)

---

### 3.2 `public.admin_permissoes`

Define quais módulos e ações cada usuário admin pode acessar. Relação 1:1 com `user_admin`.

```sql
create table public.admin_permissoes (
  id                          bigint generated by default as identity not null,
  created_at                  timestamp with time zone not null default now(),
  gerenciar_equipe_adm        boolean null,
  editar_num_wpp_suporte      boolean null,
  acessar_modulo_financeiro   boolean null,
  acessar_modulo_paciente     boolean null,
  acessar_modulo_medico       boolean null,
  validar_cadastro_medicos    boolean null,
  acessar_modulo_dashboard    boolean null,
  id_admin                    uuid not null,
  constraint admin_permissoes_id_admin_fkey
    foreign key (id_admin) references user_admin (id)
    on update cascade on delete cascade
);
```

**Colunas e significado:**
| Coluna | Controla acesso a |
|---|---|
| `gerenciar_equipe_adm` | Página "Gerenciar equipe" — convidar/remover admins e editar permissões |
| `editar_num_wpp_suporte` | Campo WhatsApp do suporte em Configurações |
| `acessar_modulo_financeiro` | Módulo completo de Gestão Financeira |
| `acessar_modulo_paciente` | Módulo de Pacientes (listagem + detalhe) |
| `acessar_modulo_medico` | Módulo de Médicos (listagem + detalhe + edição) |
| `validar_cadastro_medicos` | Ação de aprovar/recusar cadastro de médico |
| `acessar_modulo_dashboard` | Página Dashboard com KPIs e gráficos |

**Regras críticas:**
- `null` e `false` têm o **mesmo efeito: acesso negado**. Sempre tratar `!= true` como bloqueado.
- Apenas `true` explícito libera o acesso.
- Ao criar novo admin: inserir em `admin_permissoes` com **todas as colunas como `false`** por padrão.
- Apenas admins com `gerenciar_equipe_adm = true` podem alterar permissões de outros.
- Um admin **não pode remover sua própria permissão** de `gerenciar_equipe_adm` (proteção no frontend e backend).

### 3.3 Diagrama de relações

```
auth.users (Supabase Auth)
    │
    ▼
public.user (id)
    │
    ▼
public.user_admin (id_user → user.id)       [ON DELETE CASCADE]
    │
    ▼
public.admin_permissoes (id_admin → user_admin.id)  [ON DELETE CASCADE]
```

**Query padrão — carregar admin logado com permissões:**
```sql
SELECT
  u.nome,
  u.email,
  ua.id AS admin_id,
  p.gerenciar_equipe_adm,
  p.editar_num_wpp_suporte,
  p.acessar_modulo_financeiro,
  p.acessar_modulo_paciente,
  p.acessar_modulo_medico,
  p.validar_cadastro_medicos,
  p.acessar_modulo_dashboard
FROM public.user_admin ua
JOIN public.user u ON u.id = ua.id_user
LEFT JOIN public.admin_permissoes p ON p.id_admin = ua.id
WHERE ua.id_user = auth.uid();
```

## 4) Lógica de Permissões no Frontend

### 4.1 Ao carregar o app admin
1. Verificar se usuário autenticado tem registro em `user_admin`
2. Se não → redirecionar para `/acesso-negado` ou `/login`
3. Se sim → carregar `admin_permissoes` e armazenar no **contexto/estado global**
4. Usar as permissões para: exibir/ocultar itens da sidebar, habilitar/desabilitar botões e seções

### 4.2 Guards de rota por módulo

| Rota | Permissão necessária |
|---|---|
| `/dashboard` | `acessar_modulo_dashboard = true` |
| `/pacientes` | `acessar_modulo_paciente = true` |
| `/medicos` | `acessar_modulo_medico = true` |
| `/medicos/:id/aprovar` | `validar_cadastro_medicos = true` |
| `/financeiro` | `acessar_modulo_financeiro = true` |
| `/configuracoes/equipe` | `gerenciar_equipe_adm = true` |
| `/configuracoes` | Livre (seções internas têm restrição própria) |
| `/consultas` | Livre para todos os admins |
| `/notificacoes` | Livre para todos os admins |

### 4.3 Sidebar — visibilidade condicional

```
Dashboard         → visível se: acessar_modulo_dashboard = true
Pacientes         → visível se: acessar_modulo_paciente = true
Médicos           → visível se: acessar_modulo_medico = true
Gestão Financeira → visível se: acessar_modulo_financeiro = true
Consultas         → sempre visível
Notificações      → sempre visível
Configurações     → sempre visível (conteúdo interno restrito)
```

### 4.4 Comportamento de acesso negado
- **Item de sidebar:** ocultar completamente (não exibir item sem permissão)
- **Rota acessada diretamente sem permissão:** redirecionar para `/dashboard` com toast "Acesso não autorizado"
- **Botão de ação sem permissão específica:** desabilitar com tooltip "Você não tem permissão para esta ação"
- **Campo editável sem permissão:** exibir apenas em modo leitura (sem ícone de lápis)

## 5) Fluxo — Gerenciar Equipe Admin

Disponível apenas para admins com `gerenciar_equipe_adm = true`.

### 5.1 Listar equipe

```sql
SELECT
  u.nome,
  u.email,
  ua.id        AS admin_id,
  ua.created_at,
  p.gerenciar_equipe_adm,
  p.editar_num_wpp_suporte,
  p.acessar_modulo_financeiro,
  p.acessar_modulo_paciente,
  p.acessar_modulo_medico,
  p.validar_cadastro_medicos,
  p.acessar_modulo_dashboard
FROM public.user_admin ua
JOIN public.user u ON u.id = ua.id_user
LEFT JOIN public.admin_permissoes p ON p.id_admin = ua.id
ORDER BY ua.created_at ASC;
```

### 5.2 Convidar novo admin
1. Admin com permissão digita e-mail do novo membro
2. Frontend chama Edge Function `admin-invite` (com service role) → `supabase.auth.admin.inviteUserByEmail(email)`
3. Após aceite do convite e criação da conta: webhook ou trigger insere em `public.user_admin` e `public.admin_permissoes` (todas as permissões `false`)
4. Admin gerenciador define permissões via toggles na interface

### 5.3 Editar permissões de um admin

```sql
UPDATE public.admin_permissoes
SET
  gerenciar_equipe_adm      = :gerenciar_equipe_adm,
  editar_num_wpp_suporte    = :editar_num_wpp_suporte,
  acessar_modulo_financeiro = :acessar_modulo_financeiro,
  acessar_modulo_paciente   = :acessar_modulo_paciente,
  acessar_modulo_medico     = :acessar_modulo_medico,
  validar_cadastro_medicos  = :validar_cadastro_medicos,
  acessar_modulo_dashboard  = :acessar_modulo_dashboard
WHERE id_admin = :admin_id;
```

### 5.4 Revogar acesso

```sql
-- ON DELETE CASCADE remove admin_permissoes automaticamente
DELETE FROM public.user_admin WHERE id = :admin_id;
```

**UX:** exibir modal de confirmação antes de revogar. Não permitir auto-revogação.

## 6) Configurações da Plataforma

### Tabela `configuracoes` (criar se não existir)

```sql
CREATE TABLE IF NOT EXISTS public.configuracoes (
  id         uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  chave      text UNIQUE NOT NULL,
  valor      text NOT NULL,
  descricao  text,
  updated_at timestamptz DEFAULT now()
);

INSERT INTO public.configuracoes (chave, valor, descricao) VALUES
  ('whatsapp_suporte', '', 'Número do WhatsApp exibido no app para suporte'),
  ('percentual_reembolso_cancelamento', '50', 'Percentual de reembolso em cancelamentos com menos de 24h')
ON CONFLICT (chave) DO NOTHING;
```

### 6.1 WhatsApp do suporte
- **Chave:** `whatsapp_suporte`
- **Permissão para editar:** `editar_num_wpp_suporte = true`
- **Admins sem esta permissão:** veem o número mas sem ícone de edição
- **Formato de exibição:** `(99) 99999-9999`
- **Query de leitura:** `SELECT valor FROM configuracoes WHERE chave = 'whatsapp_suporte'`
- **Query de escrita:** `UPDATE configuracoes SET valor = :numero, updated_at = now() WHERE chave = 'whatsapp_suporte'`

### 6.2 Percentual de reembolso em cancelamentos
- **Chave:** `percentual_reembolso_cancelamento`
- **Permissão:** qualquer admin autenticado
- **Validação obrigatória:** inteiro entre 0 e 100 (inclusive)
- **Uso:** aplicado no cálculo quando paciente cancela consulta com menos de 24h de antecedência
- **Exibição:** campo com símbolo `%` e hint explicativo

## 7) Regras por Módulo

### 7.1 Médicos (requer `acessar_modulo_medico = true`)

**Status de plantão (`user_medico.status_plantao`):**
| Valor | Significado | Visível no app |
|---|---|---|
| `inicio_imediato` | Disponível agora | Sim — chip verde |
| `em_consulta` | Em atendimento | Sim — chip âmbar |
| `inativo` | Desativado pelo admin | Não |
| `pendente` | Aguardando aprovação | Não |

**Aprovação (requer `validar_cadastro_medicos = true`):**
1. Admin visualiza médicos com `status_plantao = 'pendente'`
2. Revisa: `crm_url`, `cnh_url`, `foto_fundo_branco_url`
3. Aprovar → `status_plantao = 'inicio_imediato'`
4. Recusar → `status_plantao = 'inativo'` (ou manter `pendente`)
5. Notificar médico por e-mail via Edge Function

### 7.2 Pacientes (requer `acessar_modulo_paciente = true`)
- Visualização completa — sem edição de dados pessoais (LGPD)
- Admin pode desativar conta em caso de violação

### 7.3 Consultas (livre)

**Status da consulta (`consulta.status`):**
| Valor | Significado |
|---|---|
| `agendada` | Marcada, não realizada |
| `realizada` | Concluída |
| `cancelada` | Cancelada |

**Status de pagamento (`consulta.status_pagamento`):**
| Valor | Significado |
|---|---|
| `pago` | Confirmado |
| `em_aberto` | Aguardando |
| `reembolso` | Devolvido ao paciente |
| `atrasado` | Em atraso |

**Regras de reembolso:**
- ≥ 24h de antecedência → reembolso de 100%
- < 24h → aplicar `percentual_reembolso_cancelamento` de `configuracoes`
- Cancelamento iniciado pelo admin → reembolso definido manualmente (0–100%)

### 7.4 Gestão Financeira (requer `acessar_modulo_financeiro = true`)
- Receita bruta = soma de `consulta.valor` onde `status_pagamento = 'pago'`
- Filtros por período (`data_hora`), médico, status
- Exportação CSV dos resultados filtrados

## 8) RLS — Políticas de Segurança

```sql
-- Verificar se o usuário logado é admin
CREATE OR REPLACE FUNCTION public.is_admin()
RETURNS boolean
LANGUAGE sql SECURITY DEFINER
AS $$
  SELECT EXISTS (
    SELECT 1 FROM public.user_admin
    WHERE id_user = auth.uid()
  );
$$;

-- Apenas admins podem ler dados de médicos no painel
CREATE POLICY "admin_acessa_medicos"
ON public.user_medico FOR SELECT
TO authenticated
USING (public.is_admin());

-- Apenas admins podem ler dados de pacientes
CREATE POLICY "admin_acessa_pacientes"
ON public.user_paciente FOR SELECT
TO authenticated
USING (public.is_admin());

-- Admin lê e edita consultas
CREATE POLICY "admin_gerencia_consultas"
ON public.consulta FOR ALL
TO authenticated
USING (public.is_admin());
```

**Regra geral:** operações sensíveis via Supabase Edge Functions com service role no servidor. Nunca expor service role no cliente.
